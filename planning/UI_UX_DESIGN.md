# 数据标注平台 · UI/UX 与页面设计

> 版本：v1（配套 `PRD.md`，作为 PRD §23 的展开）
> 定位：**知识生产工作台（Knowledge Production Workspace）**，而非传统后台管理系统
> 心智模型：`代码编辑器（Cursor/VS Code）` + `阅读器（Reader/Readwise）`，**不是** `企业 CRM 后台`

---

## 0. 设计理念与心智模型

本平台的本质是**「阅读 → 判断 → 记录」的高强度重复劳动**。一个标注员一天要处理几百上千条，每条 < 90 秒。决定成败的不是页面好不好看，而是**单位时间内能否高质量地完成更多条**。所以所有设计都服务于一个目标：

> **让人停留在「内容」上，让操作「消失」。**

五条贯穿全局的原则（与需求一一对应）：

| # | 原则 | 落地手段 | 反模式（禁止） |
|---|---|---|---|
| 1 | **沉浸式工作流** | 工作台全屏铺满；侧栏可一键折叠；用浮层/内联反馈代替弹窗 | 多级后台菜单、频繁 modal、跳回列表页 |
| 2 | **键盘优先** | Cmd+K 命令面板；高频动作单键触发；焦点态模型避免冲突 | 必须用鼠标点按钮才能提交/切换 |
| 3 | **阅读优先** | 中栏是「阅读surface」：65–75 字符行宽、1.7 行高、17px 正文 | 把正文塞进窄表格单元格 |
| 4 | **连续工作流（刷题式）** | 提交后**自动加载下一条**，内容滑入，永不回列表 | submit 后弹「成功」对话框并返回列表 |
| 5 | **长时友好** | 深色优先、低饱和、克制动效、护眼字色（非纯白） | 大面积纯白、高饱和蓝、闪烁动画 |

**两种「模式」的分区**——这是整个信息架构的根：

- **沉浸区（Workspace）**：标注工作台、审核工作台。全屏、极简外壳、无持久侧栏，键盘驱动。
- **管理区（Hub）**：数据集、看板、用户、schema 编辑。有一条极窄的左侧 Activity Rail 做导航。

进入沉浸区时，导航外壳自动隐藏；用户的视野里只剩「这一条要标什么」。

## 1. 整体信息架构

```
┌─────────────────────────────────────────────────────────────────┐
│  AUTH（无外壳）                                                    │
│   /login                                                          │
└─────────────────────────────────────────────────────────────────┘
                              │ 登录后按角色落地
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
   annotator → /workspace   admin → /datasets    reviewer → /review

┌──────────────────── HUB · 管理区（有 Activity Rail）────────────────┐
│  Activity Rail（48px，图标+tooltip，当前态高亮）                    │
│   ▸ Datasets   ▸ Annotate   ▸ Review   ▸ Dashboard   ▸ Admin       │
│   底部：用户头像 / 主题切换 / ?帮助                                 │
│                                                                    │
│  内容区：数据集 / 看板 / 用户 / schema 编辑                        │
└─────────────────────────────────────────────────────────────────┘

┌──────────────── WORKSPACE · 沉浸区（外壳自动隐藏）─────────────────┐
│  标注工作台 /workspace      审核工作台 /review/:taskId             │
│  仅一条极窄顶栏 + 三栏内容 + 底部快捷键条                          │
│  Rail 默认隐藏，hover 左缘或按 `\` 唤出                           │
└─────────────────────────────────────────────────────────────────┘
```

**导航哲学**：
- **不做面包屑 + 顶部导航条的传统组合**。导航靠两件东西：① 极窄 Activity Rail（空间换专注）；② **命令面板 Cmd/Ctrl+K**（Linear/Notion 招牌）——跳数据集、切视图、跑动作、查快捷键，全靠它。
- Rail 在 Hub 常驻，在 Workspace 自动收起。这样「管理」与「干活」两种心境物理隔离。
- 角色决定 Rail 上可见的入口：annotator 只看到 Annotate / Dashboard；reviewer 多 Review；admin 全部。

## 2. 页面树

```
/login                              登录（无外壳）
/                                   按角色重定向

── HUB（管理区，带 Rail）─────────────────────────────────────────
/datasets                           数据集列表（卡片网格）
/datasets/upload                    上传向导（首次建 dataset）
/datasets/:id                       数据集详情：进度 / 批次历史 / 概览 / 导出抽屉
/datasets/:id/schema                form_schema 编辑器（字段映射 + 实时预览）
/dashboard                          进度看板（覆盖率 + 产能，SSE 实时）
/admin/users                        用户与权限（轻量表）

── WORKSPACE（沉浸区，外壳自动隐藏）──────────────────────────────
/workspace?dataset=:id              标注工作台（claim→读→标→submit→next）
/review                             审核队列（内嵌左栏，非独立列表页）
/review/:taskId                     审核工作台（对比 + 决策）
```

> 设计取舍：**导出**折叠进 `/datasets/:id` 的右侧抽屉（一个面板，不占独立路由）；**审核队列**不做成独立列表页，而是审核工作台的左栏——reviewer 进 `/review` 即直接进入第一条待审，左栏列队列。**页面越少，跳转越少，心流越稳**。

## 3. 每个页面的目标

| 页面 | 一句话目标 | 成功指标 |
|---|---|---|
| 登录 | 3 秒进系统，零干扰 | 单字段聚焦，Enter 直登 |
| 数据集列表 | admin 一眼看清「哪些项目、进度多少、谁在标」 | 不点进去就能判断健康度 |
| 上传向导 | 上传 → READY 全程 < 10 min，每步状态可见 | 失败原因当场可读 |
| 数据集详情 | 单个项目的指挥中心：进度 / 批次审计 / schema / 导出 | 增量同步结果（new/updated）一眼可见 |
| schema 编辑器 | 把 PG 列「翻译」成标注表单，所见即所得 | 改一处右侧预览实时变 |
| **标注工作台** | **让标注员心流不断地标完一条又一条** | 单条 < 90s，全程不碰鼠标可完成 |
| 审核队列/工作台 | reviewer 快速对比、秒级裁决、批量推进 | A/R 单键裁决，自动进下一条 |
| 进度看板 | 管理者掌握全局：覆盖率、吞吐、瓶颈 | 实时刷新，无需手动刷新 |
| 用户与权限 | 轻量增删改角色 | 不喧宾夺主，一屏搞定 |

## 4. 页面布局说明

### 4.1 登录 `/login`

居中卡片，深色画布，**单列**。无插画噪音，品牌字标 + 用户名 + 密码 + 主按钮。

```
            ┌───────────────────────────────┐
            │          ◆ Labeling           │   ← 字标，Inter 600
            │                               │
            │   Username                    │
            │   [____________________]      │
            │   Password           👁        │   ← show/hide toggle
            │   [____________________]      │
            │                               │
            │   [        Sign in        ]   │   ← Enter 直接提交
            │                               │
            │   错误内联显示在按钮上方       │
            └───────────────────────────────┘
```
- 自动聚焦用户名；`Enter` 提交；提交态：按钮转 spinner，失败时红色内联文案（非弹窗）。
- `autocomplete="username/current-password"`，配合浏览器密码管理器。

### 4.2 数据集列表 `/datasets`（Hub）

卡片网格（非传统表格）。每张卡是一个「项目」，进度环 + 关键数字 + 状态色点。

```
┌─ Rail ─┬──────────────────────────────────────────────────────────┐
│ ◆      │  Datasets                      [+ New dataset]   ⌘K        │
│ ▸ Data │  ─────────────────────────────────────────────────────    │
│   Anno │  ┌──────────────┐ ┌──────────────┐ ┌──────────────┐       │
│   Revw │  │ ● READY      │ │ ◐ IMPORTING  │ │ ● READY      │       │
│   Dash │  │ 高考专业库    │ │ 论文摘要集    │ │ 客服对话      │       │
│   Admin│  │ ◔ 68%        │ │  解析中...    │ │ ◕ 91%        │       │
│        │  │ 2.3k / 3.4k  │ │              │ │ 5.0k / 5.5k  │       │
│ ───────│  │ 8 标注员在线  │ │              │ │ 3 标注员     │       │
│ 👤 ☾ ? │  └──────────────┘ └──────────────┘ └──────────────┘       │
└────────┴──────────────────────────────────────────────────────────┘
```
- 状态用**色点 + 文字**（不靠颜色单一编码）：READY 绿、IMPORTING 蓝脉冲、PAUSED 灰、FAILED 红、DONE 紫。
- 进度环（donut）+ 「已标 / 总数」tabular 数字。卡片 hover 微抬升（elevation，不位移）。
- 点卡进详情；admin 才见 `+ New dataset`。

### 4.3 上传向导 `/datasets/upload`（Hub）

**线性分步**（步骤指示器 + 可返回），不堆一个大表单：

```
①上传文件  →  ②确认源表/主键/hash列  →  ③恢复中（实时日志）  →  ④READY
```
- ① 拖拽区（.sql/.backup），显示大小上限 500MB，超限即时报错。
- ② 反射出的源表/列预览；选 `source_pk_column`、勾选 `hash_columns`（默认全列，**高亮建议排除 created_at/updated_at**）。
- ③ 恢复进度用骨架 + 流式日志行（IMPORTING）；失败把 `import_batch.error` 原文展示 + 「重试 / 删除」。
- ④ 完成跳详情，提示「下一步：编辑标注字段」。

### 4.4 数据集详情 `/datasets/:id`（Hub）

单列内容 + 右侧可滑出抽屉（导出）。顶部进度概览，下面批次审计时间线。

```
高考专业库                              [编辑 schema] [同步新版本] [导出 ▸]
● READY · v3 · 3,420 行 · hash: title,body,category
────────────────────────────────────────────────────────────────────
覆盖率  ████████████████░░░░  68%      PENDING 1090 · CLAIMED 24 · DONE 2306
────────────────────────────────────────────────────────────────────
导入批次（审计）
  ▸ #5  2026-05-20  v3.dump   new +120  updated +35   by admin
  ▸ #4  2026-05-12  v2.dump   new +800  updated  +0   by admin
  ▸ #1  2026-05-01  v1.dump   new +2500 updated  +0   by admin
```
- `new/updated` 是增量同步的核心审计，必须醒目（绿/琥珀小徽标）。
- 「同步新版本」「导出」「编辑 schema」是 admin 三大动作，置顶。
- 导出点开右侧抽屉：选 jsonl/csv + 版本分桶提示 + 流式下载进度。

### 4.5 schema 编辑器 `/datasets/:id/schema`（Hub）

**左编辑 / 右预览** 双栏，所见即所得：

```
┌─ 字段列表（拖拽排序）──────────┬─ 实时预览（即标注右栏长什么样）──┐
│ 源字段（只读，来自反射）        │  ┌────────────────────────────┐ │
│  · id        int    Input     │  │ 类别 *      [A][B][C] ◀ seg │ │
│  · text      text   TextArea  │  │ 置信度      ●━━━━○ 0.7      │ │
│ ─────────────────────────────  │  │ 备注        [            ] │ │
│ 标注字段（可增删/配置）         │  └────────────────────────────┘ │
│  + 加字段 ▾  Select/Rating... │   改左边任意一项 → 右边即时重渲染 │
└───────────────────────────────┴──────────────────────────────────┘
```
- 加标注字段时选 widget（Select/Rating/Tag/Confidence/TextArea/Relation），配 options/min/max/required。
- **破坏性改动（删字段/改类型）弹确认**，提示「旧 annotation 将标 superseded」（PRD §10.3）。其余改动直接 `form_schema_version + 1`。

### 4.6 进度看板 `/dashboard`（Hub）

数据卡 + 少量图表（克制，不堆 dashboard）。SSE 实时。

```

────────────────────────────────────────────────────────────────
各数据集覆盖率（横向条形，>5 类用 bar 不用 pie）
  高考专业库  ███████████░░░░  68%
  客服对话    ██████████████░  91%
────────────────────────────────────────────────────────────────
标注员产能（今日，tabular 数字）        近 7 日吞吐（折线）
  陈   312   ▲                          ╱╲    ╱
  李   288                            ╱╲╱  ╲╱
  王   201                          ─────────────
```
- 图表遵循 charts 规范：横向 bar 比较、折线看趋势、有图例/tooltip、空态有提示、低对比网格线。

### 4.7 用户与权限 `/admin/users`（Hub）

轻量。一张安静的表（非重边框企业表），行内改角色，软删。

```
用户              角色 ▾        创建            操作
陈晓            [annotator ▾]  2026-04-01     ⋯
李明            [reviewer  ▾]  2026-04-03     ⋯
admin          [admin     ▾]  2026-03-20     ⋯
                                              [+ 新建用户]
```
- 角色用 inline segmented/下拉直接改；危险操作（删除）在 `⋯` 溢出菜单里，与常规操作**空间隔离**，二次确认 + undo toast。

## 5. 核心组件设计

| 组件 | 作用 | 关键设计点 |
|---|---|---|
| **AppShell / ActivityRail** | 管理区导航骨架 | 48px 图标栏，当前态左侧 2px 高亮条 + 实色图标；Workspace 自动 collapse |
| **CommandPalette（⌘K）** | 全局导航/动作加速器 | 模糊搜数据集、切视图、跑动作、查快捷键；`cmdk` 库实现，键盘全程 |
| **DatasetCard** | 列表项 | 进度环 + tabular 数字 + 状态色点 + 在线人数；hover elevation |
| **ReadingPane** | 中栏阅读 surface | 居中 max 720px、17px/1.7、`text-primary` 非纯白；J/K 滚动 |
| **SchemaForm** | 右栏动态表单 | schema 驱动；卡片分组；渐进展示；单键完成字段（见 §9） |
| **FieldWidgets** | 各类标注控件 | Segmented/Rating/TagInput/Confidence/TextArea/RelationCombobox |
| **ShortcutHintBar** | 底部快捷键提示 | **随焦点态变化**只显当前可用键；不喧宾夺主 |
| **LeaseTimer** | 租约倒计时 | mono tabular；绿→琥珀(<5m,脉冲)→红(<1m)；到期阻断 |
| **AutosaveIndicator** | 草稿状态 | 圆点 + 文案：editing→saving→saved；localStorage 落盘 |
| **DiffView** | reviewer 对比 | 源↔标注 / 旧↔新 / AI↔人；变化高亮 |
| **AISuggestionChip / AIPanel** | AI 预标注（预留） | 建议以「幽灵填充」入表单；接受/改/拒 + reasoning 折叠 |
| **ProgressRing / CoverageBar** | 进度可视 | donut + 横向 bar；色不单一编码 |
| **Toast / InlineBanner** | 反馈 | 替代 modal；3–5s 自动消；危险操作带 Undo；`aria-live` |
| **EmptyState** | 空态 | 队列空、无数据集、无待审——都给「下一步」引导而非空白 |

> **组件库取舍**：放弃 Ant Design 的企业外观。建议 **headless 基元（Radix UI）+ Tailwind 自绘主题**，命令面板用 `cmdk`。动态表单见 §9 的引擎说明。

## 6. 标注工作台详细设计（核心）

### 6.1 整体布局：三栏 + 顶栏 + 底栏

```
┌─ TopBar（48px，安静）──────────────────────────────────────────────────────┐
│ 高考专业库   ·   #88123   ·   ⟳ round 2   ·   ⧗ 28:42   ·   ● saved        │
│                                              [skip ⏭]  [release ⎋]  12/340  │
├─ Left（280px，可折叠 \）──┬─ Center（flex，阅读区）──────┬─ Right（380px，表单）─┤
│ CONTEXT                  │                              │  标注                  │
│  pk        4501          │   待标注的原文标题            │  ┌─ 核心 ──────────┐  │
│  source    v3            │   ─────────────────────      │  │ 类别 *           │  │
│  imported  #5            │                              │  │ [A][B][C][D]  ◀1-4│  │
│ ───────────────────────  │   正文段落，17px / 1.7 行高，  │  │                  │  │
│ LABELS（本轮已填回显）    │   居中 65–75 字符宽，沉浸     │  │ 置信度           │  │
│  （round 1 旧值，淡显）   │   阅读。其余源列收在下方       │  │ ●━━━━━○  0.7    │  │
│ ───────────────────────  │   「详情 ▾」里，Space 展开。  │  └──────────────────┘  │
│ HISTORY                  │                              │  ┌─ 补充（可折叠）──┐  │
│  round1 by 李 · 2d ago   │   ▾ 详情（所有源列只读）      │  │ 备注 textarea    │  │
│  superseded              │     category 历史 …          │  │ [              ] │  │
│                          │                              │  └──────────────────┘  │
│                          │                              │                        │
│                          │                              │  [ 提交并下一条 ⏎ ]    │
├──────────────────────────┴──────────────────────────────┴────────────────────────┤
│ ⏎ 提交  ·  1-4 选类别  ·  Tab 下一字段  ·  Space 展开详情  ·  ⌘K 命令  ·  ? 帮助 │ ← ShortcutHintBar
└────────────────────────────────────────────────────────────────────────────────┘
```

**三栏职责**：
- **左（参考/上下文，安静）**：源行的次要元信息（pk、来源版本、批次）、本轮回显的旧标签、历史轮次与 superseded 记录。低对比、低音量——「需要时瞄一眼」。可 `\` 折叠让中栏更宽。
- **中（阅读，主角）**：把源表里**最该读的字段**（admin 在 schema 里标 primary）放大成阅读 surface；其余源列折进「详情」，`Space` 展开。这是视线停留 80% 时间的地方。
- **右（标注，行动）**：schema 驱动的表单，卡片分组（核心 / 补充），底部唯一主 CTA「提交并下一条」。

### 6.2 工作流：claim → 读 → 标 → submit → next（刷题式）

```
进入 /workspace?dataset=:id
   │  自动 POST /tasks/claim（SKIP LOCKED 抢一条 + 30min lease）
   ▼
任务滑入（自左向右 200ms）→ 自动聚焦右栏第一个字段
   │  读中栏 → 键盘填右栏（1-4 选类、Tab 走字段）
   ▼
⏎ 提交（乐观更新：右栏即清空、计数 +1）
   │  POST /tasks/:id/submit（幂等，校验 assigned_to）
   ├─ 成功 → 清草稿 → 自动 claim 下一条 → 内容滑入  ✅ 心流不断
   └─ 失败（已超时/被回收）→ 红色内联「已超时，请重新领取」+ 自动重新 claim
   ▼
队列空 → 安静空态「本数据集已标完 🎉 切换数据集？⌘K」
```

关键：**submit 后不回列表、不弹框、不刷新整页**。下一条以「替换」动画进入同一容器（crossfade，exit 快于 enter），保持空间连续。这是「连续工作流」的物理实现。

### 6.3 节奏与认知负荷优化

- **进入即可标**：claim 完成自动把焦点放到右栏第一个**必填**字段，标注员落座即开干。
- **左中右的视觉音量梯度**：左最淡、中最亮、右居中。视线天然 中→右 流动，不左右横跳。
- **lease 计时不焦虑**：平时灰绿安静；< 5min 变琥珀并轻脉冲 + 顶栏出现「延长」；< 1min 变红。心跳 60s 自动续，用户基本无感（PRD §7.2）。
- **草稿永不丢**：每次改字段 debounce 写 `localStorage draft:${taskId}`；崩溃/误关回来自动恢复并提示「已恢复草稿」。
- **错误就地说人话**：提交失败不弹框，红色内联文案 + 明确恢复路径（自动重领），符合 `error-clarity` / `error-recovery`。

## 7. Reviewer 审核工作台

Reviewer 的任务不是「填」，是「判」。所以页面**对比优先 + 秒级裁决 + 自动推进**，与标注台不同。

### 7.1 布局：队列栏 + 对比区 + 决策栏

```
┌─ 队列（240px）──┬─ 对比区（flex）────────────────────────────┬─ 决策（300px）─┐
│ 待审 86         │  源内容（只读）          标注结果（只读）    │ 裁决            │
│ ▸#88123 陈 72s  │  ┌──────────────────┐   ┌──────────────────┐│  ┌───────────┐  │
│ ·#88120 李 65s  │  │ 标题…             │   │ 类别:  历史 ✓     ││  │ A 通过     │  │
│ ·#88119 王 88s  │  │ 正文段落…         │   │ 置信度: 0.8       ││  │ R 打回重标 │  │
│ ·…              │  │                   │   │ 备注:  …          ││  └───────────┘  │
│                 │  └──────────────────┘   └──────────────────┘│ note（R 时聚焦）│
│ J/K 上下切       │  ← 重标轮次显示 旧↔新 diff，变化高亮 →       │ [          ]   │
│ A 通过 R 打回    │                                             │ E 编辑标注      │
└─────────────────┴─────────────────────────────────────────────┴────────────────┘
```

### 7.2 对比效率
- **左源右标，等高并排**，眼睛只需横扫一次即可对照「内容 vs 判断」。
- **三种 diff 场景**复用 `DiffView`：① 源 ↔ 标注（默认）；② 重标轮次 旧 ↔ 新（变化高亮）；③ 未来 AI ↔ 人。
- 不能审自己的标注（PRD §14.1），队列里自动过滤。

### 7.3 快速决策 + 批量
- `A` = approve 并自动跳下一条；`R` = needs_redo（焦点自动落到 note 输入，写完 `⏎` 确认并跳下一条）。
- `J/K` 在队列里上下移动，不离手。整套支持「连按 A、A、A」秒速过一批高质量数据。
- `E` 进入编辑态：reviewer 可直接微调标注（小修不必打回重标），保存即覆盖为有效 annotation。
- 队列空 → 「全部审完」空态。

> reviewer 与 annotator 的根本差异：标注台是**生产**（focus 在右栏输入），审核台是**判别**（focus 在中部对比 + 单键裁决）。

## 8. 快捷键体系

### 8.1 焦点态模型（避免冲突的关键）

单键快捷键和文本输入天然冲突（在备注里打字「1」不能触发选类别）。用**三种焦点态**解决：

| 焦点态 | 何时 | 单键行为 |
|---|---|---|
| **READING**（无输入聚焦） | 刚进任务、读内容时 | 全部单键生效：`1-9` `Q/W/E` `J/K` `Space` `S` 等 |
| **WIDGET**（聚焦在 Rating/Select 等非文本控件） | Tab 到选择类字段 | `1-9` 选选项；选完自动 Tab 到下一字段 |
| **FIELD**（聚焦在 text/textarea） | 在文本框打字 | **单键全禁用**；仅组合键生效（`⌘⏎` 提交、`Tab` 离开、`Esc` 失焦） |

底部 `ShortcutHintBar` **随焦点态实时切换**，永远只显「此刻能按的键」。

### 8.2 全局（任何页面）

| 键 | 动作 |
|---|---|
| `⌘/Ctrl + K` | 命令面板（搜数据集 / 切视图 / 跑动作） |
| `g` then `d` / `w` / `r` / `b` | 跳 Datasets / Workspace / Review / dashBoard |
| `?` | 快捷键速查浮层 |
| `\` | 折叠/展开侧栏 |
| `⌘/Ctrl + \` | 切换深/浅色 |

### 8.3 标注工作台

| 键 | 动作 | 条件 |
|---|---|---|
| `1`–`9` | 选中当前 Select/Rating 第 N 个选项 | READING / WIDGET |
| `Q` `W` `E` `R` … | 快速套用 admin 配置的标签映射（可自定义） | READING |
| `Tab` / `Shift+Tab` | 下一 / 上一字段 | 任意 |
| `Space` | 展开/收起中栏「详情」 | READING（FIELD 下输空格） |
| `J` / `K` | 中栏阅读区向下 / 上滚动 | READING |
| `⏎` | 提交并下一条 | READING/WIDGET（FIELD 下用 `⌘⏎`） |
| `⌘/Ctrl + ⏎` | 提交并下一条 | 任意（含 textarea 内） |
| `⌘/Ctrl + S` | 显式保存草稿（平时自动保存） | 任意 |
| `S` / `→` | 跳过（放回池，领下一条） | READING |
| `Esc` | 失焦当前输入；**已失焦时**= 释放任务回池（带 Undo toast） | — |
| `⌘/Ctrl + Z` | 撤销刚提交（lease 内重新认领该条） | — |

### 8.4 审核工作台

| 键 | 动作 |
|---|---|
| `A` | approve 并跳下一条 |
| `R` | needs_redo（焦点落 note） |
| `J` / `K` | 队列下/上移动 |
| `E` | 进入编辑标注态 |
| `⏎` | 确认当前裁决 |
| `Esc` | 退出编辑 / 关闭 note |

> 设计依据：高频动作（选类、提交、过审）落在**左手 home row 与右手 Enter** 的自然分工——左手按 1-4 选项、右手 Enter 提交，**双手不离键盘**即可完成一条。

## 9. 动态表单交互设计

### 9.1 schema 驱动渲染

后端返回 `form_schema.annotation_fields`（PRD §10.2），前端按 `widget` 映射到控件并渲染。**渲染层与 PG 类型映射沿用 PRD §10.1**，但控件实现替换为「键盘优先」的自绘版本。

**引擎建议（与 PRD 的 Antd+Formily 协调）**：保留 schema 驱动思路，但**不直接用 Antd 控件**——它们的交互（必须点开下拉、无单键选项）不满足键盘优先。两条路线：
- **A（推荐）**：`react-hook-form` + `zod`（按 schema 动态构建）+ **自绘 headless 控件层**（Radix 基元 + Tailwind）。完全掌控单键选择、segmented、焦点流。
- **B**：保留 Formily 作状态引擎，但**注册自定义组件**替换默认 Antd 渲染。

### 9.2 各字段类型的最佳交互

| 类型 | 控件 | 交互精要 |
|---|---|---|
| **rating（1–5）** | 横向 Segmented / 星级 | `1-5` 直接选；选完自动 Tab；当前值实色填充 |
| **select（枚举 ≤5）** | **Segmented 按钮组** | 一屏全列出，`1-N` 单键选，**零下拉点击** |
| **select（枚举 >5）** | Combobox + type-ahead | 输首字母过滤，↑↓ 选，`⏎` 定 |
| **multi-select / tag** | Chip 输入 | type-ahead，`⏎` 加 chip，`Backspace` 删，已选高亮 |
| **confidence（0–1）** | Slider + 预设 | 滑块 + `低/中/高` 三快捷预设 + 直接输数 |
| **textarea** | 自增高文本域 | auto-grow、字数计数、autosave、`⌘⏎` 提交 |
| **relation link** | 异步 Combobox | type-ahead 拉小表（v1 admin 手配），显示主键+标签 |
| **bool** | Toggle / 二段 Segmented | `Y/N` 或 `1/0` 单键 |

**核心理念**：**能用 Segmented 就不用下拉**。下拉每次都要「点开→看→点选」三步，Segmented 把选项常驻并支持单键，是提速关键。

### 9.3 避免「大长表单」

- **卡片分组**：`核心字段`（必填、最常用）一张卡常显；`补充字段`（可选）折进第二张可折叠卡，渐进展示（`progressive-disclosure`）。
- **必填优先排序**：进入自动聚焦第一个必填；可选项默认收起。
- **智能默认**：可配默认值/上一条沿用（如「类别」常连续相同），减少重复输入。
- **就地校验**：`onBlur` 校验（非每次击键），错误显示在字段下方（`inline-validation` / `error-placement`）。

### 9.4 提速汇总
进入自动聚焦 → 1-N 单键选 → 自动跳下一字段 → 必填填完 → `⏎` 提交 → 下一条。理想路径：**手指不离 home row，一条 < 10 秒**。

## 10. UI 状态流转

### 10.1 任务生命周期（前端视角）

```
claiming（骨架）→ loaded/editing（聚焦首字段）→ submitting（乐观，右栏清空）
   ├─ ok    → cleared → 自动 claim next → entering（滑入）
   └─ fail  → error（红内联「已超时」）→ 自动 re-claim
队列空 → empty（空态 + ⌘K 切换提示）
```

### 10.2 草稿状态

```
empty ──首次改字段──> dirty ──debounce 600ms──> saving ──> saved（● saved）
saved ──继续改──> dirty ……    submit 成功 ──> 清除 localStorage
```
- 指示器文案：`editing…` → `saving…` → `● saved`；崩溃恢复 → `已恢复草稿` 提示。

### 10.3 租约状态

```
healthy(>5min, 灰绿) ─倒计时─> warning(<5min, 琥珀+脉冲, 顶栏「延长」)
                              ─> critical(<1min, 红) ─> expired(阻断输入, 「重新领取」)
心跳 60s 自动续约 → 回 healthy
```

### 10.4 连接状态

```
online ──断网──> offline（顶栏黄条「离线，提交将排队」）──重连──> 同步排队的提交
```

### 10.5 AI 建议状态（预留）

```
idle ──请求──> suggesting（幽灵填充 + 微光）──> 用户：accept / modify / reject
accept → 值落定（标记来源 ai）  modify → 值落定（来源 ai-edited）  reject → 清空回人工
```

## 11. 深色模式设计

**深色为默认且主力**（长时护眼）。管理区可选浅色；工作台/审核台始终深色优先。用语义 token，禁止组件内写死 hex。

### 11.1 表面分层（Surface Elevation）

不靠重阴影，靠**逐级提亮的中性面**表达层级（Linear/Cursor 手法）：

| Token | Hex | 用途 |
|---|---|---|
| `--bg` | `#0C0E12` | 画布（OLED 近黑） |
| `--surface-1` | `#14171C` | 左/右栏、卡片底 |
| `--surface-2` | `#1B1F26` | 字段组、抬升卡片 |
| `--surface-3` | `#232830` | hover、popover、命令面板 |
| `--border-subtle` | `#20242B` | 分隔线（极淡） |
| `--border` | `#2C313A` | 卡片/输入边框 |
| `--border-strong` | `#3A414C` | 聚焦/强调边框 |

### 11.2 文字层级（非纯白，降眩光）

| Token | Hex | 对比(@`--bg`) | 用途 |
|---|---|---|---|
| `--text-primary` | `#E6E9EF` | ~14:1 | 正文、标题 |
| `--text-secondary` | `#A0A7B4` | ~7:1 | 次要说明 |
| `--text-tertiary` | `#6B7280` | ~4.6:1 | 元信息、占位 |
| `--text-disabled` | `#4B5563` | — | 禁用（配 opacity） |

### 11.3 语义/功能色

| Token | Hex | 含义 | 用途 |
|---|---|---|---|
| `--focus` / brand | `#818CF8`（indigo） | 焦点/选中/品牌 | 焦点环、active nav、选中选项 |
| `--success` | `#3FB950`（green） | 成功/提交 | 提交完成、READY、approve |
| `--warning` | `#E3B341`（amber） | 警示 | lease<5min、离线条 |
| `--danger` | `#F0626B`（red） | 危险/打回 | needs_redo、release、删除 |
| `--ai` | `#A78BFA`（violet） | AI 辅助 | AI 建议、reasoning（专属色，区别于 focus 蓝） |

要点（对应 `color-dark-mode` / `color-accessible-pairs`）：
- 深色用**去饱和的色调变体**，非把浅色直接反相。
- 功能色一律**配图标/文字**，不靠颜色单一传意（`color-not-only`）。
- 正文 ≥ 4.5:1、次要 ≥ 3:1，**两套主题各自验证对比**，边框/焦点态在两套都可见。
- 焦点环 2px `--focus`，**绝不 `outline:none` 不补替代**（键盘优先的命门）。

### 11.4 字体

| 角色 | 字体 | 用途 |
|---|---|---|
| UI / 正文 | **Inter**（300–700） | 界面、阅读区正文（17px/1.7） |
| 数据 / 等宽 | **JetBrains Mono** | pk、content_hash、ID、lease 计时器（tabular 防跳动）、JSON |

`font-display: swap`；正文 16–17px、行高 1.7、阅读区 measure 65–75ch。

## 12. 移动端策略

**明确结论：标注工作台与审核工作台不支持移动端，桌面键盘优先。**

理由：三栏布局 + 双手键盘流是效率核心，触屏小屏无法承载——硬塞只会让效率崩塌、认知负荷飙升，违背产品目标。

分级支持：

| 设备/宽度 | 标注/审核台 | 管理区（看板/数据集/详情） |
|---|---|---|
| 桌面 ≥1280px | ✅ 完整三栏（推荐） | ✅ 完整 |
| 笔记本 1024–1279px | ✅ 二栏（左栏默认折叠，`\` 唤出） | ✅ |
| 平板 768–1023px | ⚠️ 只读预览 + 「最佳体验请用桌面」提示 | ✅ 响应式单列 |
| 手机 <768px | ❌ 拦截页：「标注台需桌面访问」 | ✅ **只读监控**：看板/进度可查，不可操作 |

即：**手机/平板可「看进度」，不能「干标注」**。未来若有「移动快速审核」需求（A/R 两键），再单独设计精简单卡流，不复用桌面三栏。

## 13. UI/UX 设计 Rationale（为什么这样设计）

| 设计决策 | 对应需求 | 为什么 |
|---|---|---|
| 沉浸区/管理区物理分离，Rail 在工作台自动隐藏 | 沉浸式、降疲劳 | 「干活」和「管理」是两种心境，混在一起会持续分心；隐藏外壳把像素和注意力都还给内容 |
| submit 后自动加载下一条、内容滑入替换 | 连续工作流 | 回列表→找下一条→点进去是三次认知中断；刷题式消除中断，吞吐直接上去 |
| 中栏阅读 surface（65–75ch / 1.7 / 17px / 非纯白） | 阅读优先、长时友好 | 本质是「读 + 判断」，阅读舒适度直接决定持续时长和判断质量；纯白大面积是长时眩光主因 |
| 三焦点态模型 + 单键优先 + Segmented 替下拉 | 键盘优先、提速 | 下拉的「点开-看-选」三步是隐形效率杀手；单键选项 + 双手 home row 分工，把单条压到 10 秒级 |
| 命令面板 ⌘K 取代多级菜单 | 沉浸式、键盘优先 | 菜单越深越慢；模糊搜索一步直达，且天生键盘友好（Linear/Notion 已验证） |
| localStorage 草稿 + 乐观提交 + 内联错误 | 数据不丢、心流 | 高强度连续作业最怕「白干」和「弹窗打断」；草稿兜底 + 乐观更新让感知零延迟，失败再优雅回滚 |
| reviewer 台与 annotator 台异构（对比+单键裁决） | 数据质量、对比效率 | 审核是「判别」不是「生产」，并排 diff + A/R 单键 + 自动推进，让抽检不成为瓶颈 |
| 深色优先 + 语义 token + 表面分层（非重阴影） | 长时友好、专业工具感 | 连续数小时作业深色护眼；分层提亮的中性面是现代工具（Cursor/Linear）的专业质感来源，避免企业蓝边框的廉价感 |
| AI 以「幽灵填充 + 接受/改/拒 + reasoning 折叠」预留 | 后续 AI 预标注 | 把 AI 设计成「草稿建议」而非「替人决定」，自然嵌入现有表单流；human-in-the-loop 不打断、可追溯来源（ai/ai-edited/human） |
| 移动端只读、工作台拦截 | 不为伪需求买单 | 键盘三栏流无法在触屏复刻，与其做个残废版，不如明确边界、把资源压在桌面体验上 |

## 附：AI 预标注的自然融入路径（前瞻）

虽 v1 不开发，UI 已留好接口，未来零重构接入：

- **数据**：`annotations.data` 已是 JSONB，可加 `_source: 'human'|'ai'|'ai-edited'`、`_ai_confidence`、`_ai_reasoning` 元字段，无需改表结构。
- **交互**：任务加载时若有 AI 预标，右栏字段以**幽灵态（dimmed + violet 边）预填**；顶部一行 `✨ AI 已建议 · 一键全采纳(⌘A) · 查看依据(?)`。用户改任意字段即从 `ai` 转 `ai-edited`，未动则采纳为 `ai`，全清则 `human`。
- **证据**：每个 AI 建议字段旁可展开 `reasoning / evidence`（引用源文片段，与中栏内容高亮联动）。
- **审核**：reviewer 的 DiffView 天然支持 `AI ↔ 人` 第三种对比，质检 AI 产出零额外开发。

> 这样 AI 是「加速器」而非「黑箱」，与键盘优先、连续工作流完全兼容。
